## 1. 簡介

**Docker Dashboard** 是一個 Windows 桌面工具，讓你集中管理多個 Docker Compose 專案，免去在不同終端機視窗之間切換的麻煩。

**主要特色：**

- 一鍵啟動 / 停止 / 重建多個 Docker Compose 專案
- 即時查看容器日誌，支援關鍵字過濾
- 容器崩潰時系統匣氣球通知
- Git 分支一覽與快速切換
- 檔案變更自動觸發重建（Watch 功能）
- 支援 Docker Desktop 與 WSL2 兩種執行模式
- 透過 Velopack 自動推播更新

## 2. 系統需求與安裝

### 系統需求

| 項目 | 最低需求 |
|------|---------|
| 作業系統 | Windows 10 64-bit（版本 1903+）或 Windows 11 |
| .NET 執行環境 | .NET 10 Desktop Runtime |
| Docker | Docker Desktop 4.x 或 WSL2 + Docker Engine |
| 記憶體 | 256 MB（應用程式本身） |

### 安裝方式

#### 方式一：Velopack 安裝器（推薦）

1. 至 [GitHub Releases](https://github.com/bro278911/DockerDashboard/releases) 下載最新的 `DockerDashboard-win-Setup.exe`
2. 執行安裝器，程式會自動安裝至 `%LocalAppData%\DockerDashboard`
3. 安裝完成後桌面與開始選單會出現捷徑
4. 日後更新：應用程式啟動時自動檢查並提示

#### 方式二：免安裝 portable（開發用）

直接執行 `DockerDashboard.exe`（需手動安裝 .NET 10 Desktop Runtime）。

## 4. 新增與移除專案

### 新增專案

1. 點擊工具列「**匯入資料夾**」
2. 在資料夾選擇對話框中選取包含 `docker-compose.yml` 的根目錄
3. 工具會自動掃描該資料夾，支援的 compose 檔名：
   - `docker-compose.yml` / `docker-compose.yaml`
   - `compose.yml` / `compose.yaml`
   - 自動合併同目錄的 `override` 和 `build` 變體
4. 若資料夾為 Git 儲存庫，左側樹狀圖會顯示目前分支名稱

> **注意：** 掃描時自動略過 `obj/`、`bin/`、`node_modules/`、`.git/`、`packages/` 子目錄，避免誤判。

### 移除專案

1. 在左側樹狀圖上對專案資料夾按右鍵
2. 選擇「**移除**」
3. 移除的資料夾路徑會暫存於「**最近移除清單**」（最多 10 筆）

### 重新加入最近移除的專案

在左側樹狀圖下方的「最近移除」區域，點擊路徑可快速重新加入。

## 5. 啟動與停止服務

### 全部啟動

點擊「**全部啟動**」，工具會對所有已匯入的 compose 專案**並行**執行 `docker compose up -d`（`Task.WhenAll`），縮短等待時間。

按鈕條件：至少有一個服務，且並非所有服務都已在執行中。

### 全部停止

點擊「**全部停止**」，並行對所有專案執行 `docker compose down`。

按鈕條件：至少有一個服務正在執行。

### 單一服務操作

在左側樹狀圖選取某個服務後，可在右鍵選單或詳細面板看到：

| 操作 | 對應指令 |
|------|---------|
| 啟動 | `docker compose up -d <service>` |
| 停止 | `docker compose stop <service>` |
| 重啟 | `docker compose restart <service>` |
| 重建並重啟 | `docker compose up -d --build <service>` |
| 查看日誌 | `docker compose logs -f <service>` |

### 快速開啟瀏覽器

服務若有對外 port（如 `3000->3000`），在樹狀圖點選該服務後，可點擊出現的連結直接在瀏覽器開啟 `http://localhost:<port>`。

## 6. 建置與重建 Image

### 增量重建（BuildKit Cache）

```
工具列 → [增量重建]
```

使用 Docker BuildKit 的 layer cache，**只重建有變動的層**，適合日常開發迭代。速度比完整重建快數倍。

若想限制並行建置數量，可在**設定 → BuildKit 平行度**中設定（0 = 自動）。

### 完整重建（--no-cache）

```
工具列 → [完整重建]
```

加上 `--no-cache` 強制不使用任何 cache，確保完全乾淨的建置環境。適合解決「有時 cache 導致舊程式碼殘留」的問題，但耗時較長。

## 7. 查看即時日誌

### 串流日誌

在左側樹狀圖選取服務後，點擊「**查看日誌**」按鈕（或右鍵選單），右側日誌面板會開始即時顯示 `docker compose logs -f` 的輸出。

### 關鍵字過濾

在日誌面板上方的過濾輸入框輸入關鍵字，日誌面板**即時過濾**只顯示含有該關鍵字的行（不區分大小寫）。清空輸入框恢復顯示全部。

### 日誌行數上限

日誌最多保留 **5000 行**；超過時自動捨棄最舊的 500 行，避免記憶體無限增長。

### 匯出日誌

點擊「**匯出**」按鈕，選擇儲存位置，可將目前日誌（已過濾或全部）存為 `.txt` 或 `.log` 檔案。

### 停止 / 清除

| 按鈕 | 功能 |
|------|------|
| 停止 | 停止日誌串流（容器繼續執行） |
| 清除 | 清空日誌面板內容（容器繼續執行） |

## 8. Git 分支切換

### 顯示目前分支

若專案資料夾為 Git 儲存庫，左側樹狀圖專案名稱旁會顯示藍色 chip 標籤，例如：

```
📁 my-project  [ ⎇ main ]
```

若工作目錄有未提交的變更，chip 可能顯示為黃色或附加 `*` 標記。

### 切換分支

1. 對專案資料夾按右鍵 → **切換分支**
2. 在彈出的 `BranchSelectorWindow` 中選擇本地分支
3. 確認後自動執行 `git checkout <branch>`
4. 分支 chip 即時更新

### 拉取最新代碼

右鍵選單 → **Git Pull**，執行 `git pull`，完成後自動更新分支與 dirty 狀態。

## 9. 容器監控與崩潰通知

### 雙迴圈監控機制

工具採用兩種方式同時監控容器狀態：

| 機制 | 說明 |
|------|------|
| **定時輪詢** | 每隔 N 秒執行 `docker ps`，全面更新所有容器狀態 |
| **事件串流** | 持續監聽 `docker events`，即時偵測 start/stop/die/kill 等事件 |

兩者互補：輪詢保證週期性同步；事件串流確保狀態變化（如崩潰）能立即反應。

### 輪詢間隔設定

預設每 **5 秒**輪詢一次，可在「**設定 → 輪詢間隔**」中調整（建議 3～30 秒）。

### 容器崩潰通知

當某容器狀態從 `Running` 變為 `Exited` 或 `Dead` 時：

1. 日誌面板追加警告訊息：`⚠️ 容器崩潰: <name> → Exited`
2. 系統匣（右下角）彈出氣球通知：**「容器崩潰警告」**

> 即使視窗已最小化到系統匣，崩潰通知仍會出現。

## 10. Watch 自動重建

Watch 功能監控指定服務的**原始碼資料夾**，偵測到檔案變更後自動觸發重建與重啟。

### 啟用 Watch

1. 在左側樹狀圖選取服務
2. 勾選「**Watch**」核取方塊（或右鍵 → 啟用 Watch）
3. 服務旁出現 👁 圖示代表已啟用

### 防抖設定（Debounce）

避免連續存檔觸發多次重建。可在「**設定 → Watch 防抖延遲**」設定等待秒數（預設 2 秒）。

例如：設定 2 秒，表示最後一次檔案變更後，**靜止 2 秒**才觸發重建。

### 全域開關

「**設定 → 自動 Watch**」可一次啟用/停用所有 Watch 監控。

## 11. 設定頁面

點擊工具列「**設定**」開啟設定視窗。

### Docker 模式

| 模式 | 說明 |
|------|------|
| **Docker Desktop** | 直接呼叫 `docker` CLI（適合安裝了 Docker Desktop 的使用者） |
| **WSL2** | 透過 `wsl -d <distro> docker` 呼叫（適合只在 WSL2 內安裝 Docker Engine） |

切換後需重新連線，若連線失敗會彈出提示詢問是否切換回另一種模式。

### WSL2 Distro 名稱

WSL2 模式下需填入正確的 distro 名稱（預設 `Ubuntu`）。可用 `wsl -l` 查看已安裝的 distro 清單。

### 輪詢間隔（秒）

容器狀態定時更新的頻率，預設 `5` 秒。

### BuildKit 平行度

`BUILDKIT_PARALLELISM` 環境變數的值，控制 BuildKit 同時建置的 stage 數量。`0` 表示由 Docker 自動決定。

### Watch 設定

| 設定項 | 說明 |
|--------|------|
| 自動 Watch | 全域啟用/停用所有 Watch 監控 |
| Watch 防抖延遲 | 最後一次變更後等待幾秒再觸發重建（秒） |

### 設定儲存位置

設定自動儲存至 `%APPDATA%\DockerDashboard\settings.json`，重啟後保留。

## 12. 系統匣（最小化背景執行）

點擊視窗右上角的「最小化」按鈕，或直接關閉視窗（X），應用程式會縮小至**系統匣（右下角通知區）**繼續執行。

### 還原視窗

雙擊系統匣圖示，或右鍵選單 → **開啟**，即可還原視窗。

### 系統匣右鍵選單

| 選項 | 功能 |
|------|------|
| 開啟 | 還原主視窗 |
| 結束 | 完全關閉應用程式 |

> 系統匣模式下，容器監控、崩潰偵測、Watch 自動重建**仍持續執行**。

## 13. 修復工具

點擊工具列「**修復**」開啟修復視窗，提供 Docker 系統清理功能。

| 操作 | 對應指令 | 說明 |
|------|---------|------|
| 清除懸空 Image | `docker image prune -f` | 移除未使用（dangling）的 image |
| 清除未使用 Volume | `docker volume prune -f` | 移除沒有任何容器掛載的 volume |
| 系統全面清理 | `docker system prune -f` | 一次清除容器、網路、image、build cache |

> ⚠️ **警告：** 系統清理為不可逆操作，請確認不需要這些資料後再執行。

## 14. 自動更新（Velopack）

應用程式整合 **Velopack** 自動更新框架，每次啟動時在背景靜默檢查 GitHub Releases 是否有新版本。

### 更新流程

```
應用程式啟動
    ↓
背景靜默檢查 GitHub Releases
    ↓
發現新版本？
  ├── 否 → 繼續正常使用（無任何提示）
  └── 是 → 工具列出現「有新版本 x.x.x」綠色按鈕
              + 系統匣氣球通知
                  ↓
              點擊按鈕
                  ↓
              下載更新（按鈕顯示「下載中 xx%」進度）
                  ↓
              下載完成 → 自動重啟並套用更新
```

### 注意事項

- **首次安裝**：透過 Velopack 安裝器安裝後才支援自動更新；直接執行 `exe` 的 portable 模式下更新按鈕不會顯示。
- **重啟必要**：更新安裝完成後，應用程式會自動重啟，未儲存的設定變更可能遺失（設定通常即時儲存，影響不大）。
- **手動觸發**：目前無手動「立即檢查更新」按鈕，若需立即檢查可重啟應用程式。

## 15. 開發者發布指南

本節說明如何打包並發布新版本（供維護者使用）。

### 前置準備（一次性）

```powershell
# 安裝 Velopack CLI 工具
dotnet tool install -g vpk

# 確認 gh CLI 已安裝並登入 GitHub
gh auth login
```

### 發布新版本

```powershell
# 步驟 1：修改版本號
# 編輯 src/DockerDashboard/DockerDashboard.csproj
# <Version>1.1.0</Version>

# 步驟 2：發布應用程式
dotnet publish src/DockerDashboard -c Release -r win-x64 --self-contained false -o publish/

# 步驟 3：vpk 打包（產生 Releases/ 資料夾）
vpk pack `
  --packId DockerDashboard `
  --packVersion 1.1.0 `
  --packDir publish/ `
  --mainExe DockerDashboard.exe `
  --channel win

# 步驟 4：建立 GitHub Release 並上傳所有打包檔案
gh release create v1.1.0 Releases/* `
  --title "v1.1.0" `
  --notes "## 更新內容
- 新增 XXX 功能
- 修復 YYY 問題"
```

### Delta 差分更新（節省下載量）

```powershell
# 加上 --delta 產生差分包（使用者只需下載差異部分）
vpk pack `
  --packId DockerDashboard `
  --packVersion 1.1.0 `
  --packDir publish/ `
  --mainExe DockerDashboard.exe `
  --delta BestSpeed `
  --channel win
```

### GitHub Actions 自動化（CI/CD）

建立 `.github/workflows/release.yml`，在 `git tag v*` 推送時自動建置與發布：

```yaml
name: Release
on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: windows-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'

      - name: Install vpk
        run: dotnet tool install -g vpk

      - name: Publish
        run: dotnet publish src/DockerDashboard -c Release -r win-x64 --self-contained false -o publish/

      - name: Pack with Velopack
        run: |
          $version = "${{ github.ref_name }}".TrimStart('v')
          vpk pack --packId DockerDashboard --packVersion $version --packDir publish/ --mainExe DockerDashboard.exe --channel win

      - name: Upload to GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          files: Releases/*
```

使用方式：

```powershell
# 修改版本號後，打 tag 推送即觸發自動發布
git tag v1.1.0
git push origin v1.1.0
```

## 16. 常見問題 FAQ

### Q: 啟動後提示「Docker 未啟動或未安裝」

**A:** 請確認：

1. Docker Desktop 已啟動（系統匣有 Docker 圖示且顯示 Running）
2. 或 WSL2 模式下，Docker daemon 已在 distro 內啟動（`sudo service docker start`）
3. 在「設定 → Docker 模式」中確認選擇的模式與實際環境一致

若使用 WSL2 模式，建議在 `/etc/wsl.conf` 中加入：
```ini
[boot]
command = service docker start
```
使 Docker 隨 WSL2 啟動自動執行。

---

### Q: 匯入資料夾後服務列表是空的

**A:** 可能原因：

1. 資料夾內沒有 `docker-compose.yml` 等支援的檔案
2. `docker compose config` 指令失敗（語法錯誤、環境變數缺少）
3. 路徑中含有特殊字元

**排除步驟：**
- 在終端機 `cd` 到該資料夾，手動執行 `docker compose config` 確認是否有錯誤
- 檢查 `.env` 檔案是否存在（若 compose 依賴環境變數）
- 點擊「重新掃描」強制重新偵測

---

### Q: WSL2 模式連線很慢

**A:** 工具會等待最多 12 秒讓 WSL2 內的 Docker daemon 就緒（[boot] 指令需要時間）。  
若 daemon 啟動穩定較快，可調整源碼中的 `maxAttempts` 值，或在 `/etc/wsl.conf` 中確保 Docker 服務自動啟動。

---

### Q: 日誌顯示亂碼或 emoji 無法顯示

**A:** 確保系統字型支援 Unicode。建議安裝「Cascadia Code」或「Noto Sans CJK」字型。

### Q: 修復後資料遺失（Volume 被刪除）

**A:** 「清除未使用 Volume」只會刪除**沒有任何容器掛載**的 volume。  
若容器已 `down`，volume 可能被判定為未使用而刪除。  
重要資料請先備份，或使用 `docker volume ls` 確認後再執行清除。

### Q: 自動更新按鈕不出現

**A:** 可能原因：

1. 目前以 portable（非安裝）模式執行，Velopack 不會顯示更新按鈕
2. 已是最新版本，無需更新
3. 網路無法連線至 GitHub（在企業防火牆環境）

---

### Q: 更新後應用程式崩潰

**A:** 
1. 前往 `%LocalAppData%\DockerDashboard` 確認安裝目錄完整
2. 確認 .NET 10 Desktop Runtime 已安裝（執行 `dotnet --list-runtimes`）
3. 至 [GitHub Releases](https://github.com/bro278911/DockerDashboard/releases) 重新下載最新安裝器重裝
